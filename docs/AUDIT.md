## ✅ Resolvidos na v0.1.227 — crítico/alto (11, verificados)

### 1. 🔴 vault_edit corrompe conteudo quando newStr contem padroes de substituicao ($&, $1, $$)
- **Onde:** `src/agent/tools.ts:168` · _bug_ · unidade `agent`
- **Problema:** toolVaultEdit conta ocorrencias com content.split(args.oldStr) (literal, correto) mas aplica a troca com content.replace(args.oldStr, args.newStr). String.prototype.replace interpreta sequencias especiais no ARGUMENTO DE SUBSTITUICAO: '$&' (match inteiro), '$`'/'$'' (antes/depois), '$1' (grupo), '$$' (literal $). Como oldStr aqui e string (nao regex), '$1'/'$&' viram string vazia e '$$' vira um unico '$'. Se o LLM editar codigo/markdown/regex contendo '$' (ex: trocar por um trecho com `$1` de regex, ou `$$` de TeX/preco), o arquivo grava texto ERRADO silenciosamente — perda de dado, e a tool reporta sucesso. [verificado]
- **Fix:** Escapar os '$' do newStr antes do replace: const safeNew = args.newStr.replace(/\$/g, '$$$$'); content.replace(args.oldStr, safeNew). Ou usar split/join: content.split(args.oldStr).join(args.newStr) (que nao interpreta padroes).

### 2. 🔴 handleEditMessage não roteia modelos de geração nem reanexa attachments
- **Onde:** `src/views/AxxaApp.tsx:2166` · _bug_ · unidade `axxa-app`
- **Problema:** handleEditMessage (linhas 2166-2179) só despacha entre runAgentTurn e streamReply. Diferente de handleSend (linha 1604) e retryError (linha 2205), ele NÃO checa isGenerationModel(caps). Se a sessão usa um modelo de geração (imagem/áudio/vídeo) e o usuário edita a própria mensagem, o fluxo cai em streamReply/runAgentTurn que chama activeProvider.streamChat — provider de geração não implementa chat, então estoura erro cru em vez de regerar a mídia. Também descarta pendingAttachments (não passa o 2º arg), perdendo imagens/notas que estavam anexadas. [verificado]
- **Fix:** Espelhar o dispatch de handleSend: const caps = getModelCapabilities(activeProviderId, activeModel); if (isGenerationModel(caps)) await runGenerationTurn(text, caps); else if (activeMode === 'agent') await runAgentTurn(text, attachments); else await streamReply(text, attachments); — montando attachments a partir de pendingAttachments como em handleSend.

### 3. 🔴 window.open sem 'noopener' permite reverse tabnabbing
- **Onde:** `src/components/chat/LinkSafetyModal.ts:29` · _security_ · unidade `chat-modals-conversations`
- **Problema:** openExternal() faz window.open(url, "_blank") sem rel/feature 'noopener,noreferrer'. Como os links vem de respostas da IA (conteudo nao confiavel), a pagina aberta recebe window.opener e pode redirecionar a aba do Obsidian/WebView (reverse tabnabbing) e ler o referrer. Ironicamente este e o modal cuja razao de existir e seguranca de link. [verificado]
- **Fix:** Trocar por window.open(url, "_blank", "noopener,noreferrer") e setar w.opener = null como fallback; no Electron preferir o utilitario do Obsidian para abrir externamente.

### 4. 🔴 Guard data-axxaSafeWired nunca e limpo no disposer -> re-wire vira no-op
- **Onde:** `src/components/chat/LinkSafetyModal.ts:125` · _memory-leak_ · unidade `chat-modals-conversations`
- **Problema:** wireExternalLinkSafety marca a.dataset.axxaSafeWired='1' e pula ancoras ja marcadas (linha 125). O disposer (linha 142) remove o listener mas NAO apaga o dataset. Em um re-render do Markdown que reutiliza os mesmos elementos <a> (ou apos cleanup do efeito), a proxima chamada ve axxaSafeWired ainda setado e nao re-adiciona o handler: o link passa a abrir sem o modal de seguranca. Tambem cria divergencia entre o set de listeners ativos e a flag. [verificado]
- **Fix:** No disposer, alem de removeEventListener, fazer delete el.dataset.axxaSafeWired (ou el.removeAttribute('data-axxa-safe-wired')) para cada ancora processada.

### 5. 🔴 Edit textarea usa --text-on-accent sobre balao NEUTRO -> texto quase invisivel
- **Onde:** `styles/main.css:734` · _css-polish_ · unidade `css`
- **Problema:** O balao do user foi mudado pra fundo NEUTRO sutil: .axxa-bubble-user usa `background: color-mix(in srgb, var(--text-normal) 8%, transparent)` (linha 716). Mas o textarea de edicao inline ainda herda do tempo em que o balao era accent: .axxa-edit-textarea seta `color: var(--text-on-accent)` (linha 734) e o placeholder tambem `color: var(--text-on-accent)` (linha 745). --text-on-accent costuma ser branco; sobre um fundo de 8% de --text-normal (quase a cor do background) o texto digitado fica praticamente ilegivel ao editar a mensagem. Estilo morto/desatualizado apos a mudanca do balao (print #16). [verificado]
- **Fix:** Trocar `color: var(--text-on-accent)` por `color: var(--text-normal)` na .axxa-edit-textarea (linha 734) e o placeholder (linha 745) por `color: var(--text-muted)`, casando com o fundo neutro do balao.

### 6. 🔴 Mensagem de imagem pode virar content[] vazio quando o data URL nao casa o regex
- **Onde:** `src/providers/anthropic.ts:146` · _bug_ · unidade `providers-openai-anthropic-gemini`
- **Problema:** Em toAnthropicPayload, quando imageAtt.length>0 entra-se no branch que monta 'blocks'. Se o data URL nao casar o regex /^data:([^;]+);base64,(.+)$/ (ex.: data URL sem ';base64', com charset, ou URL externa que comecou com 'data' mas nao bate), nenhum bloco e empurrado (linha 156 so faz push dentro do if(match)) e, se m.content for vazio, 'blocks' fica []. converted.push({ role:'user', content: [] }) gera um user message com content array vazio — a Anthropic responde 400 'messages: at least one message is required to have non-empty content' / 'content blocks must be non-empty'. Quebra o envio inteiro em vez de degradar. [verificado]
- **Fix:** Apos montar 'blocks', se blocks.length===0 cair no fallback texto-only (push {role:'user', content: m.content || '...'}) ou nao adicionar a mensagem. Garantir que toda mensagem user tenha ao menos um bloco com conteudo.

### 7. 🔴 embedOpenAICompat assume que a API devolve uma embedding por input, sem validar count
- **Onde:** `src/rag/embeddings.ts:142` · _bug_ · unidade `rag`
- **Problema:** Em embedOpenAICompat (linhas 142-145) o resultado e parsed.data ordenado por index e mapeado pra embeddings. Nao ha verificacao de que parsed.data.length === texts.length. Se a API (OpenAI/Gemini/NIM) devolver MENOS itens (truncamento, item rejeitado, partial 200), o indexer em indexer.ts:340-351 itera `for (let j = 0; j < slice.length; j++)` e acessa embeddings[j] que pode ser undefined -> quantizeEmbedding(undefined) chama unitNormalize(undefined) e estoura em `v.length`. Pior: se a ordem por index tiver buracos, os embeddings ficam DESALINHADOS dos chunks (vetor do chunk A salvo como chunk B) corrompendo o indice silenciosamente. [verificado]
- **Fix:** Apos o map em embedOpenAICompat, validar `if (out.length !== texts.length) throw new ProviderError(...)`. Idealmente reconstruir o array por posicao de `d.index` (out[d.index] = d.embedding) e checar que nenhuma posicao ficou undefined, em vez de depender de sort+map sequencial.

### 8. 🔴 base64ToTypedArray pode lancar RangeError pra float32 (byteLength nao multiplo de 4)
- **Onde:** `src/rag/quant.ts:132` · _bug_ · unidade `rag`
- **Problema:** Em base64ToTypedArray (linhas 137-141) cria-se `new Uint8Array(binary.length)` e depois `new Float32Array(bytes.buffer)`. O construtor de Float32Array exige que o byteLength do buffer seja multiplo de 4; Int8Array nao tem essa restricao. Se um base64 de um indice float32 vier truncado/corrompido (write parcial, shard cortado) com length nao-multiplo de 4, isso lanca RangeError. Em loadIndex single-file (vectorIndex.ts:333) o throw e pego pelo catch -> indice descartado (ok). MAS em searchStreamed (vectorIndex.ts:216) base64ToTypedArray e chamado DENTRO do loop de entries sem try/catch ao redor da conversao (o try so cobre o read+JSON.parse, linhas 205-214) -> um unico emb corrompido derruba a busca inteira com excecao nao tratada. [verificado]
- **Fix:** Envolver o loop `for (const s of parsed.entries)` de searchStreamed em try/catch (ou validar `bytes.buffer.byteLength % 4 === 0` em base64ToTypedArray antes de construir o Float32Array, pulando/avisando entries invalidas).

### 10. 🔴 Keywords sao normalizadas (sem acento) mas o conteudo/titulo NAO — match silenciosamente falha em PT-BR
- **Onde:** `src/components/_shared/vaultSearch.ts:84` · _bug_ · unidade `shared-components`
- **Problema:** extractKeywords() (linha 36-45) faz .normalize('NFD').replace(/[̀-ͯ]/g,'') removendo acentos, produzindo keywords como 'funcao', 'codigo', 'reuniao'. Mas em searchVault() o conteudo (lcContent, linha 84) e o titulo (lcTitle, linha 85) so passam por .toLowerCase(), sem normalizar acentos.
